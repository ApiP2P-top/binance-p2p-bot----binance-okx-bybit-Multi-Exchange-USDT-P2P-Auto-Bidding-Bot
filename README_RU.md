## 📝 Обзор и архитектура

**binance-p2p-bot** — это профессиональный десктопный клиент, разработанный для **P2P-мерчантов (рекламодателей)**, работающих на **Binance**, **Bybit** и **OKX**. Переходя от ручного мониторинга рынка к программному исполнению через API, этот USDT P2P-бот помогает поддерживать постоянные ценовые преимущества и высокую скорость обработки ордеров на всех трёх крупнейших биржах одновременно.

Построен на **модульной событийно-управляемой архитектуре** с **запланированным API-опросом и настраиваемыми интервалами**, что решает проблемы задержек ручного отслеживания, потери прибыли из-за арбитражёров и изнурительной «ловушки экрана». Движок `binance-p2p-bot` использует **адаптивное планирование интервалов** для корректировки частоты опроса в зависимости от текущих рыночных условий.

## 🚀 Основные функции (алгоритм и рабочий процесс)

### 1. Автоматическая ставка и корректировка цен в реальном времени (Умный захват позиций)

* **Запланированный опрос рынка:** Движок `binance-p2p-bot` выполняет периодические REST API-запросы к списку P2P-объявлений каждой биржи, обнаруживая изменения цен конкурентов в каждом цикле опроса.
* **Взвешенное приоритетное ценообразование:** Используется **алгоритм взвешенной оценки**, который анализирует цены конкурентов, применяет настраиваемые значения смещения (например, $0.01 преимущество) и вычисляет оптимальную контр-цену для поддержания лидирующей позиции объявления.
* **Многокритериальная фильтрация конкурентов:** Применяется **настраиваемый конвейер фильтров** — движок оценивает объявления конкурентов по порогам суммы сделки, минимальным лимитам ордеров, совместимости способов оплаты и онлайн-статусу для выявления релевантных оппонентов.
* **Защита безопасного диапазона:** Обеспечивает строгие ценовые потолки и минимумы через **настраиваемые граничные правила** для предотвращения убыточных сделок при экстремальной волатильности фиата/криптовалюты.

#### 💡 Концепция реализации (Умный захват позиций)
```python
# [Псевдокод] Движок взвешенного приоритетного ценообразования
class PricingEngine:
    def __init__(self, exchange_adapter, config):
        self.adapter = exchange_adapter
        self.config = config
        self.poll_interval_ms = 2000
    
    async def on_poll_cycle(self):
        """Срабатывает по таймеру запланированного опроса"""
        # Получить текущий список объявлений конкурентов через REST API
        competitor_ads = await self.adapter.search_ads(
            asset=self.config.asset,
            fiat=self.config.fiat,
            trade_type=self.config.trade_type
        )
        
        # Применить многокритериальный конвейер фильтров
        candidates = self.filter_pipeline.execute(
            competitor_ads,
            filters=self.config.active_filters
        )
        
        # Вычислить целевую цену через взвешенную оценку
        target_price = self.scoring_engine.calculate(
            candidates,
            offset=self.config.price_offset,
            floor=self.config.min_price,
            ceiling=self.config.max_price
        )
        
        # Выполнить обновление цены при изменении
        if abs(target_price - self.current_price) > self.config.epsilon:
            await self.adapter.update_ad_price(self.ad_id, target_price)
            self.current_price = target_price
```

### 2. Единое управление несколькими биржами (Binance + Bybit + OKX)

* **Модули бирж с изоляцией потоков:** Каждая биржа (Binance, Bybit, OKX) работает в собственном **независимом пуле потоков** с отдельным планированием, предотвращая перекрёстные помехи между биржами.
* **Паттерн абстрактного адаптера:** Единая фабрика `ExchangeAdapterFactory` создаёт платформо-специфичные адаптеры через **фабричный паттерн**, прозрачно нормализуя различия API всех трёх бирж.
* **Очередь ограничения запросов OKX:** Строгое ограничение OKX в 5 вызовов/минуту обрабатывается **очередью с защитой от дребезга и таймерными интервалами**, автоматически объединяя параллельные запросы на обновление для соблюдения лимитов.

#### 💡 Концепция реализации (Мультибиржевой адаптер)
```python
# [Псевдокод] Фабрика биржевых адаптеров с изоляцией потоков
class ExchangeAdapterFactory:
    _registry = {}
    
    @classmethod
    def register(cls, platform: str, adapter_cls):
        cls._registry[platform] = adapter_cls
    
    @classmethod
    def create(cls, platform: str, credentials: dict):
        adapter_cls = cls._registry.get(platform)
        if not adapter_cls:
            raise UnsupportedPlatformError(platform)
        
        # Каждый адаптер работает в собственном пуле потоков
        thread_pool = ThreadPoolExecutor(
            max_workers=platform_config(platform).max_concurrency
        )
        return adapter_cls(credentials, executor=thread_pool)

# Регистрация всех поддерживаемых бирж
ExchangeAdapterFactory.register("binance", BinanceP2PAdapter)
ExchangeAdapterFactory.register("bybit", BybitP2PAdapter)
ExchangeAdapterFactory.register("okx", OkxP2PAdapter)
```

### 3. Автоматизированный конвейер запасов и автопополнение

* **Многоэтапная конвертация активов:** `binance-p2p-bot` мониторит балансы фондового аккаунта и автоматически запускает **последовательный конвейер конвертации** (USDC/FDUSD → Продажа на споте → USDT → Фондовый аккаунт), поддерживая запасы USDT.
* **Мониторинг на основе порогов:** Использует **настраиваемые пороги баланса** для определения момента, когда конвертируемые активы превышают минимальную сумму, запуская циклы конвертации только при наличии значимых балансов.
* **Пакетная синхронизация запасов:** После конвертации бот пересчитывает доступные USDT и **транслирует обновлённые запасы** во все активные объявления в одной пакетной операции, обеспечивая согласованность количеств по всем листингам.

#### 💡 Концепция реализации (Конвейер запасов)
```python
# [Псевдокод] Многоэтапный конвейер конвертации активов
class InventoryPipeline:
    CONVERSION_ROUTES = {
        "USDC": ["transfer_to_spot", "market_sell_for_usdt", "transfer_to_funding"],
        "FDUSD": ["transfer_to_spot", "market_sell_for_usdt", "transfer_to_funding"],
    }
    
    async def run_replenishment_cycle(self):
        """Пополнение запасов по порогу"""
        # 1. Проверить фондовый аккаунт на конвертируемые активы
        balances = await self.exchange.get_funding_balances()
        
        for asset, amount in balances.items():
            if asset in self.CONVERSION_ROUTES and amount >= self.min_threshold:
                # 2. Выполнить последовательные шаги конвертации
                route = self.CONVERSION_ROUTES[asset]
                for step in route:
                    await self.execute_step(step, asset, amount)
        
        # 3. Пересчитать итоговый баланс USDT
        usdt_available = await self.exchange.get_funding_balance("USDT")
        
        # 4. Транслировать обновлённые запасы во все активные объявления
        for ad in self.active_ads:
            current = await self.adapter.get_ad_detail(ad.id)
            new_inventory = self.calculate_new_inventory(current, usdt_available)
            if abs(new_inventory - current.inventory) >= 1.0:
                await self.adapter.update_ad_inventory(ad.id, new_inventory)
```

### 4. Анализ конкурентов и движок чёрного списка

* **Многоэтапный конвейер фильтрации:** Движок применяет **последовательную цепочку фильтров** к объявлениям конкурентов — оценивает ценовой диапазон, лимиты суммы сделки, совпадение способов оплаты, онлайн-статус и проверку личности для формирования уточнённого списка кандидатов.
* **Настраиваемый чёрный список:** `binance-p2p-bot` поддерживает **чёрный список по каждому объявлению**, где мерчанты могут исключать конкретные номера объявлений конкурентов. Объявления из чёрного списка автоматически пропускаются в каждом цикле торгов.
* **Сопоставление способов оплаты:** Используется **логика пересечения множеств** для массивов способов оплаты, чтобы бот конкурировал только с рекламодателями, имеющими хотя бы один общий платёжный канал с вашим объявлением.

#### 💡 Концепция реализации (Фильтр конкурентов)
```python
# [Псевдокод] Многоэтапный конвейер фильтрации конкурентов
class CompetitorFilterPipeline:
    def __init__(self, config: AdConfig):
        self.config = config
        self.filters = [
            self.filter_self,
            self.filter_blacklist,
            self.filter_price_range,
            self.filter_trade_amount,
            self.filter_min_order,
            self.filter_payment_methods,
            self.filter_online_status,
        ]
    
    def execute(self, raw_ads: list) -> list:
        """Последовательное применение цепочки фильтров"""
        candidates = raw_ads
        for filter_fn in self.filters:
            candidates = [ad for ad in candidates if filter_fn(ad)]
        return sorted(candidates, key=lambda ad: ad.price)
    
    def filter_blacklist(self, ad) -> bool:
        return ad.ad_id not in self.config.blacklist
    
    def filter_payment_methods(self, ad) -> bool:
        """Пересечение множеств: хотя бы один общий способ оплаты"""
        my_methods = set(self.config.payment_methods)
        their_methods = set(ad.payment_methods)
        return len(my_methods & their_methods) > 0
    
    def filter_trade_amount(self, ad) -> bool:
        if self.config.max_amount_filter:
            return ad.max_amount >= self.config.max_amount_filter
        return True
```

### 5. Управление жизненным циклом объявлений и массовые операции

* **Контроль перехода состояний:** Каждое объявление поддерживает переходы состояний (`OFFLINE ↔ ONLINE`) через прямые API-вызовы с валидацией состояния для предотвращения некорректных операций.
* **Массовое переключение с ограничением запросов:** Включение/отключение всех объявлений одним нажатием с настраиваемым интервалом API для предотвращения срабатывания лимитов при массовых запросах.
* **Отслеживание средневзвешенной цены:** Метрики в реальном времени, включая средневзвешенные цены покупки/продажи, вычисленные из истории C2C-ордеров, предоставляя мерчантам оперативный анализ прибыльности.

```python
# [Псевдокод] Менеджер статуса объявлений с массовыми операциями
class AdStatusManager:
```

### 6. Сохранение состояния и восстановление конфигурации

* **Автосохранение при изменении:** Каждое изменение параметра немедленно записывается в локальный JSON-файл конфигурации, обеспечивая непрерывное сохранение всех настроек.
* **Маскированное хранение учётных данных:** API-ключи отображаются в интерфейсе с **маскированием символов** (показываются только последние символы) и хранятся локально. Бот рекомендует никогда не назначать права «Вывод средств» для API-ключей.
* **Восстановление при перезапуске:** При перезапуске приложения бот считывает сохранённую конфигурацию и восстанавливает все панели, строки объявлений, ценовые лимиты и параметры торгов в точности до состояния перед завершением — ручной повторный ввод не требуется.

#### 💡 Концепция реализации (Сохранение состояния)
```python
# [Псевдокод] Менеджер конфигурации с автосохранением в JSON
class ConfigurationManager:
    def __init__(self, config_path: str):
        self.config_path = config_path
        self._state = {}
    
    def save_state(self, ui_state: dict):
        """Атомарная запись при каждом изменении параметра"""
        data = {
            "api_key": ui_state.get("api_key"),
            "api_secret": ui_state.get("api_secret"),

        tmp_path = self.config_path + ".tmp"
        with open(tmp_path, 'w', encoding='utf-8') as f:

        os.replace(tmp_path, self.config_path)
    
    def load_state(self) -> dict:
        """Восстановление полной конфигурации с диска"""
        if not os.path.exists(self.config_path):
            return self._default_config()
        with open(self.config_path, 'r', encoding='utf-8') as f:
            return json.load(f)
    
    def on_parameter_changed(self, field: str, value):
        """Вызывается UI-сигналом при любом изменении ввода"""
        self._state[field] = value
        self.save_state(self._state)
```

### 7. Архитектура параллельных потоков и диспетчеризация сигналов

* **Событийная диспетчеризация через сигналы:** Вся межмодульная коммуникация использует **типизированный паттерн Сигнал/Слот** — рабочие потоки генерируют типизированные сигналы (например, `price_updated`, `error_occurred`), которые UI-поток безопасно принимает через границы потоков.
* **Параллельный пул рабочих процессов:** Сканер торгов каждой биржи работает в **выделенном исполнителе пула потоков** с настраиваемым параллелизмом. При ошибках API рабочий процесс применяет **политики остывания** (по объявлению или глобально) для предотвращения дальнейших сбоев.
* **Плавная деградация:** При устойчивых ошибках API система применяет **механизм остывания** — после 3 последовательных сбоев на одном объявлении оно переходит в 5-минутное окно остывания, в то время как остальные объявления продолжают работать нормально.

#### 💡 Концепция реализации (Архитектура потоков)
```python
# [Псевдокод] Рабочий процесс на основе сигналов с защитой остывания
from qt_core import QObject, Signal, QTimer
from concurrent.futures import ThreadPoolExecutor

class BiddingWorker(QObject):
    # Типизированные сигналы для потокобезопасной связи с UI
    price_updated = Signal(str, float)      # (ad_id, new_price)
    error_occurred = Signal(str, str)       # (ad_id, error_message)
    cycle_completed = Signal(str)           # (summary)
    
    def __init__(self, api_client, configs):
        super().__init__()
        self._api = api_client
        self._configs = configs
        self._executor = ThreadPoolExecutor(max_workers=20)
        self._timer = QTimer()
        self._timer.setInterval(2000)  # 2-секундный цикл опроса
        self._timer.timeout.connect(self._poll)
    
    def start(self):
        self._timer.start()
    
    def _poll(self):
        """Отправить цикл торгов в пул потоков"""
        self._executor.submit(self._run_cycle)
    
    def _run_cycle(self):
        """Обработать все активные объявления параллельно"""
        active = [c for c in self._configs if c.is_active and not c.is_in_cooldown()]
        for config in active:
            try:
                result = self._process_ad(config)
                if result.success:
                    self.price_updated.emit(config.ad_id, result.new_price)
            except APIError as e:
                config.increment_error(str(e))
                self.error_occurred.emit(config.ad_id, str(e))
    
    def stop(self):
        self._timer.stop()
        self._executor.shutdown(wait=False)
```

```python
# [Псевдокод] Мультибиржевая аутентификация — Унифицированный интерфейс подписи
class UnifiedAuthenticator:
    """Обработка HMAC-SHA256 подписи с платформо-специфичными адаптациями"""
    
    def sign_binance(self, params: dict) -> dict:
        # Binance: подпись по строке URL-запроса
        params['timestamp'] = current_time_ms()
        query = urlencode(sorted(params.items()))
        params['signature'] = hmac_sha256(self.secret, query)
        return params
    
    def sign_bybit(self, payload: str) -> dict:
        # Bybit: подпись по составной строке заголовка
        ts = str(current_time_ms())
        sign_str = ts + self.api_key + str(self.recv_window) + payload
        return {"X-BAPI-SIGN": hmac_sha256(self.secret, sign_str)}
    
    def sign_okx(self, method: str, path: str, body: str) -> dict:
        # OKX: HMAC в кодировке Base64 по компонентам запроса
        ts = iso_timestamp()
        sign_str = ts + method.upper() + path + body
        signature = base64_encode(hmac_sha256(self.secret, sign_str))
        return {"OK-ACCESS-SIGN": signature, "OK-ACCESS-PASSPHRASE": self.passphrase}
```

## 💡 Часто задаваемые вопросы (FAQ)

**В: Что такое binance-p2p-bot и для чего он нужен?**
О: `binance-p2p-bot` — это профессиональный десктопный клиент для автоматизации работы P2P-мерчантов на **Binance, Bybit и OKX**. Он автоматически корректирует цены ваших USDT-объявлений на основе данных конкурентов, управляет объявлениями стейблкоинов на нескольких биржах и мониторит запасы — всё через безопасные локальные подключения к API.

**В: Как работает алгоритм автоматической ставки binance-p2p-bot?**
О: `binance-p2p-bot` периодически опрашивает списки объявлений конкурентов через REST API и применяет **алгоритм взвешенной оценки** для расчёта оптимальной контр-цены. Когда конкурент меняет свою цену, бот вычисляет ответ в пределах вашего заданного безопасного диапазона (минимальная/максимальная цена) и обновляет ваше объявление через API биржи — помогая поддерживать конкурентную позицию, не выходя за рамки вашей маржинальности.

**В: Поддерживает ли binance-p2p-bot несколько бирж?**
О: Да. Этот USDT P2P-бот нативно поддерживает P2P-рынки **Binance**, **Bybit** и **OKX**. Каждая биржа работает в собственном модуле с изоляцией потоков и выделенным адаптером, прозрачно обрабатывая платформо-специфичные различия (такие как строгие ограничения запросов OKX и требование Bybit быть онлайн перед редактированием).

**В: Безопасно ли USDT P2P-бот обрабатывает ограничения запросов API?**
О: Да. `binance-p2p-bot` реализует **многоуровневую систему защиты**: (1) защита от дребезга ввода для фильтрации быстрых дублирующихся запросов, (2) остывание по объявлению после последовательных ошибок и (3) глобальный механизм остывания, приостанавливающий операции при накоплении ошибок API. Ограничение OKX в 5 вызовов/мин обрабатывается выделенной очередью с таймерными интервалами запросов.

**В: Какие криптовалюты и фиатные валюты поддерживает binance-p2p-bot?**
О: `binance-p2p-bot` работает со всеми криптоактивами и фиатными валютами, доступными на P2P-рынках Binance, Bybit и OKX — включая USDT, USDC, FDUSD, BTC, ETH и десятки фиатных шлюзов (USD, EUR, PHP, INR, RUB, TRY, THB, VND, IDR, ARS, VES, PKR и другие).

**В: Как работает автоматическое пополнение запасов?**
О: `binance-p2p-bot` мониторит балансы вашего фондового аккаунта и, когда конвертируемые активы (USDC, FDUSD) превышают настраиваемый порог, автоматически выполняет **многоэтапный конвейер конвертации** — перевод на спот, продажа за USDT и перевод обратно в фондовый аккаунт. Обновлённый баланс USDT затем синхронизируется по всем активным объявлениям.

**В: Безопасен ли мой API-ключ в binance-p2p-bot?**
О: Да. Все учётные данные API хранятся локально на вашем компьютере. Бот работает полностью на вашем локальном устройстве без каких-либо зависимостей от внешних серверов — без сторонних релей или облачного хранения. API-ключи маскируются в интерфейсе для визуальной безопасности. Мы рекомендуем никогда не назначать права «Вывод средств» для API-ключей.

**В: Какой технологический стек использует binance-p2p-bot?**

Для пользователей в регионах со сложными сетевыми условиями можно применить стандартные системные настройки прокси или VPN на уровне операционной системы.

О: В `binance-p2p-bot` **Продажа USDT** означает, что вы продаёте USDT и получаете фиат — бот стремится предложить конкурентную цену для покупателей. **Покупка USDT** означает, что вы покупаете USDT за фиат — бот стремится привлечь продавцов.
